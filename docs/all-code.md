
# Nova ERP - الموسوعة البرمجية والتوثيقية النهائية (النسخة المكتملة والمصلحة)

هذا المستند هو "الصندوق الأسود" لنظام Nova ERP. يحتوي على شرح تفصيلي للمنطق البرمجي، هيكل البيانات، وخرائط العلاقات المعقدة، مع تضمين كافة الإصلاحات البرمجية والرقابية لضمان استقرار النظام.

---

## 🏗️ القسم 1: المعمارية الهيكلية (System Architecture)

يعتمد النظام على Next.js 14 (App Router) مع Firebase كمحرك بيانات لحظي. التفاعل بين "الميدان" (موقع المشروع) و"المكتب" (المحاسبة) يتم عبر **أتمتة القيود**.

```text
src/
├── ai/                     # محركات الذكاء الاصطناعي (Gemini 1.5 Flash)
├── app/                    # المسارات الذكية (App Router)
│   ├── dashboard/          # مركز العمليات (محاسبة، مقاولات، موارد بشرية)
├── components/             # المكونات الرسومية (UI)
│   ├── accounting/         # محرك القيود والسندات
│   ├── construction/       # محرك الـ WBS والمستخلصات والزيارات
│   ├── hr/                 # محرك الرواتب والتدقيق
├── services/               # العقل المنطقي (الرواتب، التواريخ، الإشعارات)
├── lib/                    # الأنماط (Types) والأدوات (Utils)
└── hooks/                  # خطافات جلب البيانات التحليلية اللحظية
```

## 💾 القسم 3: طبقة البيانات (Data Schema)

### 3.1 - التعريفات الرئيسية (lib/types.ts)
هذا هو القاموس البرمجي الذي يحدد هوية البيانات:

*   **العميل (Client)**: الجذر الرئيسي، لا حركات بدون ملف عميل.
*   **المعاملة (Transaction)**: تمثل الخدمة، وهي مركز الربحية الأساسي.
*   **القيد (JournalEntry)**: الضابط المالي، يضمن توازن ميزان المراجعة.
*   **المشروع (Project)**: الكيان التنفيذي المرتبط بجدول كميات (BOQ).

## 💰 القسم 9: المنطق المحاسبي التفصيلي (Accounting Engine)

### 9.1 - شجرة الحسابات (COA)
تتبع ترميزاً خماسياً دولياً لسهولة استخراج القوائم المالية:
- **1**: الأصول (Asset)
- **2**: الالتزامات (Liability)
- **4**: الإيرادات (Income)
- **5**: المصاريف (Expense)

### 9.2 - سلاسل التفاعلات التلقائية (The Smart Chains)
هذا هو "ذكاء" Nova ERP؛ حيث تؤدي الأفعال إلى نتائج مالية فورية:

1.  **اعتماد العقد** ➔ إنشاء قيد مديونية للعميل (Debit: Client / Credit: Revenue).
2.  **استلام بضاعة (GRN)** ➔ زيادة المخزون وإثبات مديونية المورد (Debit: Inventory / Credit: Vendor).
    - **ملاحظة رقابية**: في حال اختلف مبلغ التوريد عن مبلغ الأمر (PO)، يعتمد النظام "مبلغ الاستلام الفعلي" في كافة التقارير المالية للمشروع لضمان دقة الربحية.
3.  **تأكيد زيارة ميدانية** ➔ تحديث الـ WBS ➔ اقتراح مستخلص (Payment Application).
4.  **صرف راتب** ➔ إغلاق العهد أو البنك وإثبات مصروف الموظف (Debit: Payroll Exp / Credit: Bank).

### 9.3 - حماية البيانات (Saving Guard)
تم تطبيق نمط `useRef` مع `useState` في كافة نماذج الإدخال:
- يمنع الحفظ المزدوج (Double Click).
- يغلق الشاشة فور بدء التوجيه الناجح (Optimistic UI).
- يظهر مؤشر تحميل (`Loader2`) لمنع التفاعل أثناء الحفظ.

## 📦 القسم 11: الرقابة المخزنية (Inventory Control)

### 11.1 - فحص الرصيد اللحظي (Stock Guard)
يمنع النظام إجراء أي "مردود مشتريات" أو "تسوية تلف" بكمية أكبر من الرصيد المتوفر في المستودع المختار، وذلك عبر محرك `fetchStockBalances` الذي يحلل كافة الحركات السابقة لحظياً.

## 🔗 القسم 14: خريطة العلاقات (The Master Map)

| الكيان المصدر | الكيان الهدف | نوع العلاقة | حقل الربط | الأثر عند الحذف |
| :--- | :--- | :--- | :--- | :--- |
| العميل (Client) | المعاملة (Tx) | 1:N | `clientId` | منع (Restrict) إذا وجد قيد مالي |
| المعاملة (Tx) | القيد (JE) | 1:N | `transactionId` | منع الحذف لسلامة المراجعة |
| المشروع (Project) | مركز الربحية | 1:1 | `auto_profit_center` | ربط P&L آلي |
| المادة (Item) | الكفالة (Warranty) | 1:1 | `itemId` | تتبع آلي من تاريخ الصرف |

---
**ملاحظة ختامية**: هذا النظام مصمم ليكون "رقيباً"؛ أي حركة غير متوازنة (مثل صرف مواد لمشروع بدون عقد) تطلق تنبيهاً فورياً للمدير لضمان سلامة الأصول والتدفقات النقدية.
