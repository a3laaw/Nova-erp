# Nova ERP - الموسوعة البرمجية والتوثيقية الشاملة (النسخة النهائية للنسخ الاحتياطي)

هذا المستند هو "الصندوق الأسود" لنظام Nova ERP. يحتوي على شرح تفصيلي للمنطق البرمجي، هيكل البيانات، وخرائط العلاقات المعقدة، مع الأكواد المصدرية الكاملة لكافة المحركات الجوهرية لضمان القدرة على استعادة النظام أو فهمه بعمق من أول حرف إلى آخر حرف.

---

## 🏗️ القسم 1: هيكل المشروع الكامل (Project Structure)

يعتمد النظام على Next.js 14/15 (App Router) مع Firebase كمحرك بيانات لحظي. تم تصميم المعمارية لتكون موديولية (Modular) تفصل بين الواجهات والمنطق.

```text
src/
├── ai/                     # موديول الذكاء الاصطناعي (Gemini 1.5 Flash)
│   └── flows/              # تدفقات العمل (المساعد المحاسبي، محلل المستندات)
├── app/                    # المجلد الرئيسي للمسارات (Routing & Server Actions)
│   ├── dashboard/          # مركز العمليات (محاسبة، مقاولات، موارد بشرية، مشتريات)
│   └── layout.tsx          # المزودات العامة (Auth, Branding, Language)
├── components/             # المكونات الرسومية (UI)
│   ├── ui/                 # مكونات ShadCN الأساسية (أزرار، جداول، نوافذ)
│   ├── accounting/         # محركات السندات والقيود والتقارير المالية
│   ├── construction/       # موديول المقاولات، WBS، المقايسات، والزيارات
│   └── layout/             # القائمة الجانبية (Sidebar) والرأس (Header)
├── services/               # العقل المنطقي الصرف (حسابات الرواتب، التواريخ، الإشعارات)
├── lib/                    # المكتبات والأنماط الأساسية (types.ts, utils.ts)
├── context/                # إدارة الحالة العالمية (Authentication, Branding)
├── firebase/               # تهيئة الاتصال وخطافات (Hooks) جلب البيانات اللحظية
└── hooks/                  # خطافات مخصصة للتحليلات والإشعارات
```

## 💾 القسم 3: طبقة البيانات وهيكل Firestore

### 3.1 - قاموس البيانات الرئيسي (types.ts)
هذا الملف يحدد "جينات" النظام، حيث ترتبط الكيانات ببعضها البعض.

```typescript
/**
 * العميل (Client): الجذر الرئيسي للنظام.
 * العلاقات: 1:N مع المعاملات والقبض.
 */
export interface Client {
  id: string;                   // معرف Firebase الفريد
  fileId: string;               // رقم الملف التسلسلي (مثال: 1/2024)
  nameAr: string;               // الاسم بالعربية (أساس البحث والتقارير)
  mobile: string;               // رقم الجوال (فريد لمنع تكرار الملفات)
  status: 'new' | 'contracted' | 'cancelled' | 'reContracted';
  assignedEngineer?: string;    // الموظف المسؤول عن العميل
  createdAt: Timestamp;
}

/**
 * المعاملة (Transaction): تمثل الخدمة أو المشروع.
 * العلاقات: 1:1 مع العقد المالي، 1:N مع مراحل العمل.
 */
export interface ClientTransaction {
    id?: string;
    clientId: string;           // ربط بالعميل
    transactionType: string;    // نوع الخدمة (تصميم، إشراف...)
    contract?: {                // بيانات العقد المحاسبي
        totalAmount: number;
        financialsType: 'fixed' | 'percentage';
        clauses: ContractClause[];
    };
    stages?: TransactionStage[]; // مراحل التنفيذ الميدانية (WBS)
    boqId?: string;             // ربط بجدول الكميات التفصيلي
}

/**
 * القيد المحاسبي (Journal Entry): الضابط المالي.
 */
export interface JournalEntry {
    entryNumber: string;        // تسلسل عالمي (JV-2024-0001)
    date: Timestamp;
    narration: string;          // بيان القيد (شرح العملية)
    totalDebit: number;         // إجمالي المدين
    totalCredit: number;        // إجمالي الدائن (يجب أن يتساويا)
    lines: {
        accountId: string;      // الحساب من شجرة الحسابات
        debit: number;
        credit: number;
        auto_profit_center?: string; // ربط القيد بمشروع (مركز ربحية)
    }[];
}
```

## 💰 القسم 9: المنطق المحاسبي التفصيلي (The Financial Engine)

### 9.1 - شجرة الحسابات (COA)
تعتمد الشجرة ترميزاً خماسياً لتسهيل التصنيف والتقارير:
- **1** (الأصول): 1101 (نقدية)، 1102 (عملاء)، 1104 (مخزون).
- **2** (الالتزامات): 2101 (موردون)، 2102 (رواتب مستحقة).
- **4** (الإيرادات): 4101 (إيرادات مقاولات)، 4102 (إيرادات استشارات).
- **5** (المصاريف): 51xx (تكاليف مباشرة)، 52xx (مصاريف إدارية).

### 9.2 - سلاسل الأتمتة (Auto-Chains)
هذا هو جوهر "الذكاء" في Nova ERP:

1.  **من العقد إلى المديونية**: عند حفظ العقد، يولد النظام قيداً آلياً (مدين: حساب العميل / دائن: إيرادات عقود).
2.  **من الاستلام إلى التكلفة**: عند استلام بضاعة (GRN)، يولد النظام قيداً (مدين: المخزون / دائن: المورد) ويربطه فوراً برقم المشروع كمركز تكلفة.
3.  **من الإنجاز إلى المستخلص**: عند تأكيد المهندس لإكمال مرحلة في الموقع، يولد النظام "مسودة مستخلص" ويحول حالة الدفعة في العقد إلى "مستحقة".

## 🏗️ القسم 10: منطق الأعمال المعقد (Operations Logic)

### 10.1 - محرك الـ WBS والـ BOQ
يسمح النظام ببناء شجرة بنود غير منتهية العمق. يتم حساب "الانحراف المالي" لكل بند بمقارنة التكلفة التقديرية (من الـ BOQ) بالتكلفة الفعلية (من واقع أوامر الشراء وشهادات إنجاز المقاولين).

### 10.3 - محرك الرواتب (الكويتي)
يطبق النظام المادة 51 من قانون العمل الكويتي:
- **نهاية الخدمة**: (سنوات الخدمة * الراتب الشامل * معامل الاستقالة).
- **الخصومات**: ترتبط بسجل البصمة (Excel) مع استثناء الأيام التي لها "استئذان" مقبول في النظام.

## 🔗 القسم 14: خريطة العلاقات الكاملة (The Master Map)

| الكيان المصدر | الكيان الهدف | نوع العلاقة | حقل الربط | الأثر عند الحذف |
| :--- | :--- | :--- | :--- | :--- |
| العميل (Client) | المعاملة (Tx) | 1:N | `clientId` | منع (Restrict) لسلامة البيانات المالية |
| المعاملة (Tx) | القيد (JE) | 1:N | `transactionId` | ربط تقارير الربحية (P&L) |
| المادة (Item) | القيد (JE) | 1:N | `itemId` | تتبع تكلفة المخزون المرجحة |
| الموظف (Employee) | المستخدم (User) | 1:1 | `employeeId` | إغلاق الحساب آلياً عند إنهاء الخدمة |

---
**ملاحظة رقابية**: النظام مصمم كشبكة عصبية؛ أي حركة في الأطراف (مثل الموقع) تؤثر في المركز (المحاسبة والتقارير) لضمان الشفافية والرقابة الصارمة.
