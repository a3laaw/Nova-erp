# خطة التحول لنظام الـ Multi-Tenancy والاستضافة الخارجية

هذه هي خارطة الطريق لتحويل Nova ERP إلى منصة سحابية تخدم مئات الشركات بشكل منفصل وآمن.

---

## المرحلة 1: العزل المنطقي (Logical Separation)
**الهدف**: فصل بيانات الشركات داخل نفس قاعدة البيانات لضمان عدم تداخل الحسابات.

1.  **تحديث الجداول**: إضافة حقل `companyId` لكافة المستندات (تم البدء فيه في تحديث الأنماط).
2.  **تحديث هوية المستخدم**: ربط كل مستخدم بـ `companyId` محدد عند الإنشاء.
3.  **تعديل محرك البحث (Fuse.js)**: تعديل الخطافات (Hooks) لإضافة فلتر `where('companyId', '==', user.companyId)` بشكل آلي قبل عرض أي بيانات.

---

## المرحلة 2: التخصيص والاستضافة الخارجية (BYO Hosting)
**الهدف**: السماح للشركات بامتلاك بياناتها في سحابة Google الخاصة بها.

1.  **محرك الاتصال الديناميكي**:
    *   إنشاء مشروع Firebase "رئيسي" (Admin Project) لتخزين إعدادات الشركات.
    *   عند طلب العميل للدخول، يتم جلب `firebaseConfig` الخاص بشركته من المشروع الرئيسي.
    *   يتم عمل `initializeApp(companyConfig)` في الواجهة الأمامية.
2.  **النطاقات المخصصة (Custom Domains)**:
    *   ربط كل شركة بنطاق فرعي (مثلاً: `arab-builders.nova-erp.com`).
    *   توجيه النطاق إلى خادم النظام الذي سيقوم بتغيير "الهوية البصرية" والاتصال بقاعدة البيانات الصحيحة بناءً على الرابط.

---

## المرحلة 3: قواعد الأمان الصارمة (Firestore Rules)
**الهدف**: منع أي اختراق للبيانات بين الشركات.

*   تحديث ملف `firestore.rules` ليشمل الشرط التالي:
```javascript
match /clients/{clientId} {
  allow read, write: if request.auth.token.companyId == resource.data.companyId;
}
```

---

## الخلاصة
بهذه الخطة، سيتحول النظام من "برنامج" إلى "بيئة عمل" (Eco-system). الشركة الصغيرة يمكنها الاشتراك في النسخة الموحدة، والشركة الكبرى يمكنها شراء النسخة وربطها بهوستنج خاص بها، مع استمرار حصول الجميع على تحديثات الكود من المصدر الرئيسي.
